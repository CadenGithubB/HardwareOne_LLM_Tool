Q: What is Hardware One?
A: Hardware One is a self-contained IoT device built on the ESP32-S3 with 8MB PSRAM. It runs sensors, networking, automation, and an on-device language model entirely locally.

Q: What processor does Hardware One use?
A: Hardware One runs on the ESP32-S3 dual-core Xtensa LX7 at 240MHz with 8MB PSRAM.

Q: What makes Hardware One different from other IoT devices?
A: Hardware One runs completely standalone with no cloud dependency. All processing, storage, and inference happen on the device itself.

Q: Does Hardware One need internet to work?
A: Hardware One does not require internet. All features run locally. Internet is optional for NTP time sync and OTA updates.

Q: What operating system does it run?
A: Hardware One runs on FreeRTOS through the ESP-IDF framework with Arduino compatibility. There is no Linux or traditional OS.

Q: How much RAM does Hardware One have?
A: Hardware One has 512KB internal SRAM plus 8MB PSRAM. The PSRAM holds the LLM model weights and KV cache during inference.

Q: What is PSRAM used for?
A: Hardware One uses PSRAM to store the LLM model weights, KV cache, and large runtime buffers. It provides 8MB of additional memory beyond the 512KB internal SRAM.

Q: How do I check available memory?
A: Hardware One shows memory info when you run the meminfo command. It shows free heap, free PSRAM, minimum free heap since boot, and stack usage per task.

Q: What filesystem does Hardware One use?
A: The filesystem uses LittleFS on internal flash for settings, web assets, and small files. An optional SD card provides external storage for maps, logs, and model files.

Q: Where are settings stored?
A: Settings are stored in LittleFS on internal flash at /system/settings.json. They persist across reboots and firmware updates.

Q: How do I export my settings?
A: Settings can be exported by running settingsexport from the CLI or using the Settings page in the web interface to download a JSON file.

Q: How do I import settings?
A: Settings can be imported by running settingsimport from the CLI or uploading a settings JSON file through the web Settings page.

Q: What WiFi modes does Hardware One support?
A: WiFi supports station mode to connect to an existing router and access point mode to create its own network. Both can run simultaneously during setup.

Q: What is bonded mode?
A: WiFi bonded mode disables the access point so the device only connects as a station to a trusted network. The AP broadcast stops but WiFi remains active.

Q: How do I connect to WiFi?
A: WiFi is configured through the first-time setup wizard, the wificonnect SSID PASSWORD CLI command, or the web Settings page.

Q: Why does WiFi only work on 2.4GHz?
A: WiFi on the ESP32-S3 only supports 2.4GHz 802.11 b/g/n. The 5GHz band is not supported by the chip hardware.

Q: How do I check WiFi signal strength?
A: WiFi signal strength is shown by running wifistatus to see RSSI in dBm. Values above -60 are strong, -60 to -75 are moderate, below -80 is weak.

Q: What happens if WiFi disconnects?
A: WiFi automatically attempts to reconnect. Sensor polling, automations, and the CLI continue to work without WiFi.

Q: How do I scan for nearby networks?
A: WiFi scanning is done with the wifiscan command. It lists all visible 2.4GHz networks with their SSID, channel, RSSI, and encryption type.

Q: What is the WiFi access point password?
A: WiFi AP password is set during first-time setup. You can change it in the Settings page or with the appassword CLI command.

Q: Can I hide the access point SSID?
A: WiFi access point broadcasts its SSID by default. Disabling SSID broadcast is not currently supported.

Q: What is ESP-NOW?
A: ESP-NOW is a peer-to-peer wireless protocol for direct communication between ESP32 devices. It requires no router or access point.

Q: How does ESP-NOW differ from WiFi?
A: ESP-NOW sends packets directly between ESP32 devices without any router. It has lower latency but shorter range than infrastructure network connections.

Q: How do I pair ESP-NOW devices?
A: ESP-NOW pairing is done by adding the peer device MAC address with espnowadd MAC_ADDRESS. Both devices must be on the same radio channel. Run espnowstatus to verify.

Q: What is the range of ESP-NOW?
A: ESP-NOW typically reaches 10-30 meters indoors through walls and up to 100 meters outdoors with clear line of sight.

Q: Does ESP-NOW use TCP/IP?
A: ESP-NOW uses its own lightweight frame protocol directly over the 802.11 radio. It operates below the TCP/IP stack with lower overhead and latency.

Q: Can I send data to multiple ESP-NOW peers?
A: ESP-NOW supports multiple peers. Add multiple peer MAC addresses and broadcast to all of them. Each peer must be registered with espnowadd.

Q: What is BLE?
A: Bluetooth Low Energy is a short-range wireless protocol for communicating with nearby devices like phones and other Bluetooth peripherals.

Q: How do I enable Bluetooth?
A: Bluetooth is enabled by running bleenable from the CLI or toggling Bluetooth in the web Settings page. BLE advertising starts automatically once enabled.

Q: What is the BLE range?
A: Bluetooth Low Energy typically works within 5-15 meters indoors. Range depends on obstacles and the receiving device antenna quality.

Q: Can I use BLE and WiFi at the same time?
A: Bluetooth and the 2.4GHz radio can run concurrently on the ESP32-S3. The firmware manages time-division between them automatically.

Q: What is MQTT?
A: MQTT is a lightweight publish-subscribe messaging protocol. Hardware One publishes sensor data to a broker and subscribes to control topics.

Q: How do I configure MQTT?
A: MQTT is configured by setting the broker address, port, username, and password in the web Settings page or with the mqttconfig CLI command. Default port is 1883.

Q: What data does MQTT publish?
A: MQTT publishes sensor readings including temperature, humidity, pressure, distance, GPS coordinates, and device status. Each sensor publishes on its own subtopic.

Q: What MQTT topic format does Hardware One use?
A: MQTT topics follow the pattern prefix/sensor_type. The prefix is configurable in settings. Example: hardwareone/bme280/temperature.

Q: Does MQTT support TLS?
A: MQTT connects over plain TCP on port 1883 by default. TLS support depends on the firmware build configuration.

Q: What happens if the MQTT broker is unreachable?
A: MQTT retries connection automatically. Sensor polling and local features continue without interruption. Messages are not queued indefinitely to prevent memory exhaustion.

Q: What is the BME280?
A: The BME280 is an environmental sensor that measures temperature, humidity, and barometric pressure over I2C.

Q: What temperature range does the BME280 measure?
A: The BME280 measures -40C to 85C with an accuracy of plus or minus 1C. Self-heating may cause slightly elevated readings if polled frequently.

Q: How accurate is the BME280 humidity reading?
A: The BME280 humidity accuracy is plus or minus 3% relative humidity in the 20-80% range. Accuracy decreases at extremes.

Q: Can the BME280 measure altitude?
A: The BME280 measures barometric pressure which can be converted to approximate altitude. Accuracy depends on weather conditions and calibration.

Q: How do I read BME280 data?
A: The BME280 is read by running bme from the CLI to see current temperature, humidity, and pressure. The web dashboard also shows live readings.

Q: What is the ICM-42670-P?
A: The ICM-42670-P is a 6-axis inertial measurement unit that provides acceleration and angular velocity readings over I2C.

Q: What does the IMU measure?
A: The ICM-42670-P IMU measures linear acceleration and rotational velocity on X, Y, Z axes. It detects motion, tilt, orientation, and vibration.

Q: How do I calibrate the IMU?
A: The ICM-42670-P is calibrated by running imucal with the device on a flat, stable surface. The calibration stores offsets for zero-bias correction.

Q: What is the VL53L1X?
A: The VL53L1X is a time-of-flight laser distance sensor. It measures distance to an object by timing a 940nm laser pulse reflection.

Q: What is the range of the VL53L1X?
A: The VL53L1X measures from 4cm to approximately 400cm. Best accuracy is within 200cm. Performance varies with target surface reflectivity.

Q: Does ambient light affect the VL53L1X?
A: The VL53L1X is relatively immune to ambient light because it uses a narrow 940nm laser and measures pulse return time rather than reflected intensity.

Q: What is the MLX90614?
A: The MLX90614 is a non-contact infrared thermometer that measures surface temperature of objects without touching them.

Q: What is the range of the MLX90614?
A: The MLX90614 measures surface temperatures from -70C to 380C. The sensor itself operates between -40C and 125C with plus or minus 0.5C accuracy around room temperature.

Q: Does the MLX90614 measure distance?
A: The MLX90614 only measures infrared thermal radiation. It reports the temperature of the object in its field of view but not the distance to it.

Q: What is the APDS9960?
A: The APDS9960 is a multifunctional sensor that detects ambient light, color, proximity, and hand gestures over I2C.

Q: What gestures can the APDS9960 detect?
A: The APDS9960 detects up, down, left, and right hand swipe gestures within approximately 10cm of the sensor.

Q: What does the APDS9960 proximity reading mean?
A: The APDS9960 proximity value ranges from 0 to 255 where higher values indicate a closer object. It detects objects within roughly 10cm.

Q: Does Hardware One have GPS?
A: GPS is available as an optional module providing latitude, longitude, altitude, speed, and satellite count. Data is parsed from NMEA sentences over serial.

Q: How long does GPS take to get a fix?
A: GPS cold start takes 30-60 seconds outdoors with clear sky view. Warm start with cached almanac data takes 5-15 seconds. GPS does not work indoors.

Q: What is the GPS accuracy?
A: GPS accuracy is typically 2-5 meters outdoors with good satellite visibility. Accuracy degrades near buildings, trees, and other obstructions.

Q: How do I read GPS data?
A: GPS data is read by running gps from the CLI to see current coordinates, altitude, speed, heading, and satellite count. The web dashboard shows GPS on the map tab.

Q: What is the PCA9685?
A: The PCA9685 is a 16-channel PWM driver that controls servo motors and other PWM devices over I2C. It generates precise PWM signals independently from the main processor.

Q: How many servos can the PCA9685 control?
A: The PCA9685 controls up to 16 servos simultaneously. Each channel can be set to a different angle or duty cycle independently.

Q: How do I control a servo?
A: Servo control is done by running servo CHANNEL ANGLE where channel is 0-15 and angle is 0-180. The PCA9685 must be detected on the I2C bus first.

Q: What are NeoPixels?
A: NeoPixel LEDs are addressable RGB LEDs that can be individually set to any color. Hardware One controls them through a data pin with precise timing.

Q: How many NeoPixels can Hardware One drive?
A: NeoPixel strips support up to 256 LEDs on a single data line. Longer strips need adequate power supply. The firmware supports color, brightness, and animation patterns.

Q: How do I set NeoPixel colors?
A: NeoPixel colors are set by running led COLOR to set all LEDs or led INDEX COLOR to set a specific LED. Colors can be hex values like FF0000 for red or named colors.

Q: What is the OLED display?
A: The OLED is a 128x64 pixel monochrome display connected over I2C. It shows status information, sensor readings, menus, and the LLM chat interface.

Q: How do I navigate the OLED menu?
A: The OLED menu is navigated using the physical buttons to scroll through menu items and select options. The display shows the current mode, sensor readings, and network status.

Q: What is the automation system?
A: Automations run user-defined rules that trigger actions based on sensor readings, time schedules, or device events. Rules are configured through the web interface or CLI.

Q: How do I create an automation rule?
A: Automations are created on the Automations page in the web interface. Define a trigger condition, optional time constraint, and an action.

Q: What triggers can automations use?
A: Automations support sensor thresholds, time schedules, device events, and received messages on subscribed topics.

Q: What actions can automations perform?
A: Automations can send messages, control servos, set NeoPixel colors, trigger alerts, toggle outputs, run CLI commands, and chain to other automation rules.

Q: How many automation rules can I create?
A: Automations support typically 20-30 rules without issues. Each rule uses a small amount of heap memory for its condition and action.

Q: What is OTA?
A: OTA is Over-The-Air firmware update. It replaces the running firmware with a new version uploaded over the network without needing a USB cable.

Q: How do I perform an OTA update?
A: OTA updates are performed by uploading the firmware binary through the web interface OTA page or using the otaupdate CLI command. The device verifies the binary, writes it, and reboots.

Q: What happens if an OTA update fails?
A: OTA rollback is automatic. The device uses dual partitions so a failed write never bricks the device. It falls back to the previous firmware.

Q: What is the CLI?
A: The CLI is a command-line interface accessible over USB serial at 115200 baud or through the web terminal at /terminal. It provides direct control of all device features.

Q: How do I get help on CLI commands?
A: The CLI help system is accessed by running help to see all available commands grouped by category. Run help COMMAND for detailed usage of a specific command.

Q: What is the web dashboard?
A: Hardware One serves a web dashboard over the network providing real-time sensor readings, maps, file management, settings, automation configuration, and the LLM chat interface.

Q: How do I access the web dashboard?
A: Hardware One web dashboard is accessed by connecting to the same network and opening the device IP address in a browser. In AP mode, connect to the device network and browse to 192.168.4.1.

Q: What tabs does the web dashboard have?
A: Hardware One web dashboard has tabs for live sensor data, GPS map, file management, configuration, automation rules, CLI access, and language model chat.

Q: How do I upload files?
A: Files are uploaded using the Files page in the web dashboard to internal storage or the SD card. Files can also be transferred via the CLI with file transfer commands.

Q: How do I download files from the device?
A: Files are downloaded using the Files page in the web dashboard. Click on a file to download it through the browser.

Q: What file formats does the SD card support?
A: The SD card uses FAT32 filesystem. It stores map tiles, log files, LLM model files, and any other user data. Maximum tested card size is 32GB.

Q: What is the on-device LLM?
A: The language model is a tiny transformer that runs entirely on the ESP32-S3 using PSRAM. It answers questions about Hardware One without any internet connection.

Q: How big is the LLM model?
A: The language model narrow2 variant uses approximately 4.7MB for INT8 weights plus 1.3MB for the KV cache at context length 64, totaling about 6.9MB PSRAM.

Q: How fast is LLM inference?
A: The language model generates typically 2-8 tokens per second depending on model size and context length. Smaller models with shorter context are faster.

Q: How do I load an LLM model?
A: The language model is loaded from the LLM page in the web dashboard or by running llmload MODEL_PATH from the CLI. Models are stored in /system/llm/ or /sd/llm/.

Q: What is the LLM context length?
A: The language model context length determines how many tokens it can process at once. Default is 64 tokens. Longer context uses more PSRAM for the KV cache.

Q: What format are LLM model files?
A: The language model uses LLM1 binary format with magic 0x4C4C4D31. Models are converted from PyTorch using the esp32-llm-converter tool with INT8 quantization.

Q: How do I train a custom LLM model?
A: The language model is trained using the train_tiny_model_gpu.py script with your training data. Then convert the output with the esp32-llm-converter to LLM1 INT8 format.

Q: What is INT8 quantization?
A: The language model uses INT8 quantization to reduce each weight from 32-bit float to 8-bit integer with a per-group scale factor. This cuts model size by 4x with minimal quality loss.

Q: What is the KV cache?
A: The language model KV cache stores key and value tensors from previous tokens so the model does not recompute them. Larger cache allows longer context but uses more PSRAM.

Q: Can the LLM answer questions about other topics?
A: The language model is trained specifically on Hardware One documentation. It only knows about this device and its features.

Q: What authentication does Hardware One use?
A: Hardware One uses username and password authentication to protect the web interface. Credentials are set during first-time setup and stored locally.

Q: How do I change my password?
A: Settings for the password are changed by running passwd from the CLI or through the web Settings page under User Management.

Q: How do I add another user?
A: Settings for users are managed by running useradd USERNAME PASSWORD from the CLI or adding users through the web Settings page.

Q: What is NTP?
A: Hardware One uses NTP (Network Time Protocol) to sync its clock with an NTP server for accurate time in logs and automations.

Q: How do I set the timezone?
A: Settings for timezone are configured in the Settings page or with the timezone CLI command. The device uses UTC internally and applies the offset for display.

Q: What is I2C?
A: I2C is a two-wire serial bus used to communicate with sensors, the OLED display, and the PCA9685 PWM driver. Multiple devices share the same SDA and SCL lines.

Q: How do I scan for I2C devices?
A: I2C scanning is done by running i2cscan to probe all addresses on the bus. Common addresses: BME280 at 0x76, OLED at 0x3C.

Q: What pull-up resistors does I2C need?
A: I2C requires pull-up resistors on SDA and SCL lines, typically 4.7K ohm. Many breakout boards include these already. Without pull-ups the bus will not communicate.

Q: What is the I2C bus speed?
A: I2C runs at standard 100KHz speed by default. Some sensors support 400KHz fast mode but 100KHz ensures maximum compatibility.

Q: What happens if two I2C devices have the same address?
A: I2C address conflicts cause both devices to respond simultaneously, corrupting data. Each device on the bus must have a unique address.

Q: How do I check debug logs?
A: Hardware One debug logs are checked by running debugstatus to see active categories. Enable a category with debugCATEGORY 1 and view output on the serial console.

Q: What debug categories are available?
A: Hardware One provides debug categories for each subsystem including sensors, networking, LLM load, LLM tokenizer, LLM forward, LLM generate, LLM memory, I2C, automation, and more.

Q: How do I see task stack usage?
A: Hardware One shows task info when you run taskinfo. It lists all FreeRTOS tasks with their priority, state, and remaining stack space.

Q: What baud rate does the serial console use?
A: The CLI uses 115200 baud, 8N1. The ESP32-S3 USB CDC provides a virtual serial port without needing an external UART adapter.

Q: Can I use Hardware One as a datalogger?
A: Hardware One supports datalogging. Configure sensor polling intervals and enable logging to the SD card. Logs are stored as timestamped files that can be downloaded later.

Q: How often can sensors be polled?
A: Hardware One sensors can be polled every 1-10 seconds. The BME280 needs about 30ms per reading and the VL53L1X takes 20-50ms per measurement.

Q: What is the maximum SD card size?
A: The SD card supports FAT32 up to 32GB. Larger cards may work if formatted as FAT32 but are not officially tested.

Q: How do I format the SD card?
A: The SD card should be formatted as FAT32 on a computer before inserting. The device can also format it with the sdformat CLI command but this erases all data.

Q: What is deep sleep?
A: Hardware One supports deep sleep, a low-power mode where most circuits shut down. It can wake on timer, GPIO pin change, or touchpad input.

Q: How do I restart the device?
A: Hardware One is restarted by running reboot from the CLI or pressing the physical reset button. The device saves settings before restarting.

Q: What is a factory reset?
A: Hardware One factory reset clears all settings, credentials, user accounts, and returns the device to first-time setup mode. Run factoryreset from the CLI. SD card files are not affected.

Q: How do I check the firmware version?
A: Hardware One firmware version is shown by running version from the CLI or checking the bottom of the web dashboard.

Q: What is the build configuration?
A: The build system uses feature flags in System_BuildConfig.h. Set a flag to 1 to include a feature and rebuild.

Q: How do I build the firmware?
A: Building uses the ESP-IDF build system. Run idf.py build from the project directory and flash with idf.py flash. Requires ESP-IDF v5.x installed.

Q: What is the partition table?
A: The build system partition table defines how flash memory is divided between the application, OTA, LittleFS, NVS, and other partitions. It is configured at build time.

Q: How much flash does the firmware use?
A: The build system produces a firmware binary typically 1.5-2.5MB depending on enabled features. The remaining flash is used for OTA, LittleFS, and NVS storage.

Q: What is NVS?
A: Hardware One uses NVS (Non-Volatile Storage), a key-value store in flash used by ESP-IDF for calibration data and small persistent values.

Q: Can I use Hardware One without sensors?
A: Hardware One works without any sensors attached. The web interface, CLI, file management, and LLM features are all available without sensors.

Q: What voltage does Hardware One run on?
A: Hardware One operates on 3.3V. Development boards include a voltage regulator for USB 5V input. Battery input through a LiPo charger provides 3.7V nominal.

Q: How do I check battery level?
A: Hardware One battery level is shown by running battery from the CLI. The web dashboard also shows battery status when a battery monitor is connected.

Q: What is the operating temperature range?
A: Hardware One operates from -40C to 85C at the chip level. Practical outdoor range with standard components is -10C to 60C.

Q: Is Hardware One waterproof?
A: Hardware One is not waterproof. For outdoor deployment use a weatherproof IP65 or better enclosure.

Q: How do I connect sensors to Hardware One?
A: I2C sensors connect using SDA and SCL pins plus 3.3V power and ground. The GPS connects via serial UART. Check pin assignments in the build configuration.

Q: What happens during first-time setup?
A: The setup wizard creates a WiFi access point, launches a web interface, and walks you through setting a username, password, network credentials, and basic configuration.

Q: Can I skip the setup wizard?
A: The setup wizard can be skipped by connecting via USB serial and configuring settings manually with CLI commands. Run setupcomplete to mark first-time setup as done.

Q: What map tiles does Hardware One support?
A: GPS map tiles are displayed offline from the SD card. Tiles follow the standard z/x/y directory structure. Download tiles from OpenStreetMap for your area.

Q: How do I download map tiles?
A: GPS map tiles are downloaded using a tile downloader tool on your computer. Copy the tile directory to the SD card. The web Map tab loads tiles from /sd/maps/.

Q: What is the map tile cache?
A: GPS map tile cache keeps recently accessed tiles in memory for faster rendering. Cache size is limited by available RAM. Tiles are loaded from SD card on cache miss.

Q: Can Hardware One record GPS tracks?
A: GPS track recording saves timestamped coordinate files on the SD card. Tracks can be downloaded and viewed in mapping software.

Q: What is the web terminal?
A: The CLI web terminal at /terminal provides command-line access through the browser. It works like a serial console but does not require a USB cable.

Q: How do I see real-time sensor data?
A: Hardware One shows live sensor readings on the web Dashboard tab that update automatically. The CLI commands bme, imu, tof, gps show current values on demand.

Q: What refresh rate does the dashboard use?
A: Hardware One web dashboard polls sensor data every 2-5 seconds by default. Faster polling increases network traffic and CPU usage.

Q: Can I control Hardware One from my phone?
A: Hardware One web interface is responsive and works on mobile screens. Open the device IP address in your phone browser. No app installation required.

Q: Does Hardware One support multiple simultaneous users?
A: Hardware One web server handles requests sequentially on a single thread. Multiple users can connect but requests are processed one at a time.

Q: What is the maximum number of web connections?
A: Hardware One web server supports up to 7 concurrent connections by default. Additional connections are queued until a slot opens.

Q: How do I see what files are on the device?
A: Files are listed by running ls /system/ for internal storage or ls /sd/ for SD card from the CLI. The web Files page provides a graphical file browser.

Q: Can I delete files from the web interface?
A: Files can be deleted, uploaded, downloaded, and browsed through the web Files page for both internal storage and the SD card.

Q: What log files does Hardware One create?
A: Files created for logging include sensor data logs, GPS track files, and system event logs. Log files are stored on the SD card in timestamped directories.

Q: How do I clear old log files?
A: Files can be deleted through the web Files page or by running rm /sd/logs/FILENAME from the CLI. There is no automatic log rotation by default.

Q: What is the watchdog timer?
A: Hardware One uses a watchdog timer that monitors task execution and resets the device if a task stops responding. The default timeout is several seconds.

Q: Why does the device restart unexpectedly?
A: Hardware One may restart due to watchdog timeout from a stuck task, stack overflow, out of memory, or a firmware crash. Check the serial console for the reset reason.

Q: How do I interpret a crash dump?
A: Hardware One crash dumps show the PC register, call stack, and exception cause. Use the idf.py monitor tool with ELF file to decode addresses into function names.

Q: What is the boot sequence?
A: Hardware One boots by running the bootloader, starting the application, initializing hardware, mounting filesystems, starting networking, loading settings, and starting the web server and CLI.

Q: How long does boot take?
A: Hardware One typically boots in 2-5 seconds from power on to full operation. Network connection adds 1-3 seconds.

Q: What is safe mode?
A: Hardware One safe mode starts with minimal features to allow recovery from bad settings. Hold the boot button during startup to enter safe mode.

Q: Can I use both internal storage and SD card?
A: The filesystem supports both internal LittleFS at /system/ and SD card at /sd/ simultaneously. Settings use internal storage. Large files like models and maps use the SD card.

Q: What SD card speed does Hardware One support?
A: The SD card interface uses SPI mode. Typical read speeds are 1-4 MB/s depending on the card class.

Q: How do I check if the SD card is mounted?
A: The SD card status is checked by running sdstatus from the CLI. The web Files page also shows SD card presence and available space.

Q: What is the LLM sentence limit?
A: The language model sentence limit stops generation after a set number of sentences. Default is 2 sentences for Q&A to prevent rambling beyond a useful answer.

Q: What is the hard token cap?
A: The language model hard cap stops generation after a fixed number of tokens regardless of sentence boundaries. Default is 80 tokens as a safety limit.

Q: What is temperature in LLM generation?
A: The language model temperature controls randomness of token sampling. Lower values like 0.3 produce more predictable answers. Higher values like 1.0 produce more varied output.

Q: What is top-p sampling?
A: The language model top-p sampling considers only the most likely tokens whose cumulative probability exceeds the threshold. Default is 0.9.

Q: What is repetition penalty?
A: The language model repetition penalty reduces the probability of tokens that appeared recently. Default is 1.3. Higher values more strongly discourage repeated words.

Q: What is Mirostat sampling?
A: The language model Mirostat 2 is an adaptive sampling method that targets a specific surprise level in bits. It automatically adjusts the sampling distribution.

Q: What is dynamic temperature?
A: The language model dynamic temperature adjusts sampling temperature based on model confidence at each step. Off by default for Q&A models.

Q: Why should I leave dynamic temperature off?
A: The language model works best with flat temperature for Q&A. Dynamic temperature adds randomness when the model is uncertain which can produce wrong answers.

Q: How do I unload the LLM model?
A: The language model is unloaded by clicking Unload in the web LLM page or running llmunload from the CLI. This frees all PSRAM used by weights and KV cache.

Q: Can I switch between different LLM models?
A: The language model can be switched by unloading the current model then loading a different one. Only one model can be loaded at a time due to PSRAM constraints.

Q: Where do I put LLM model files?
A: The language model files go in /sd/llm/ on the SD card or /system/llm/ on internal storage. The LLM page model dropdown lists all available files from both locations.
